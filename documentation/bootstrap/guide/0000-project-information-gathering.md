# Recopilación de Información del Proyecto

## Por Qué Esta Información es Indispensable para tu Documentación

Antes de crear cualquier documento estratégico de tu proyecto, necesitas entender completamente tu contexto. Sin esta información fundamental, los documentos de tu proyecto estarán basados en suposiciones incorrectas.

**Con información completa:**
- Puedes personalizar cada documento de tu proyecto a tu realidad
- Las recomendaciones serán relevantes y aplicables a tu producto
- La alineación entre documentos de tu proyecto será natural y coherente
- El proceso de creación de documentación será eficiente y enfocado

**Sin información completa:**
- Los documentos de tu proyecto serán genéricos y desconectados
- Las recomendaciones no aplicarán a tu contexto específico
- Habrá retrabajo y correcciones constantes en tu documentación

## Recopilación de Información para tu Documentación del Proyecto

Este proceso te guiará con las siguientes preguntas estructuradas para comprender completamente tu proyecto y crear documentación de primer nivel:

### Sección A: Identificación y Propósito

- "¿Cuál es el nombre de tu proyecto?"
- "¿Podrías describir tu proyecto en una oración clara y concisa?"
- "¿Qué tipo de proyecto es? (SaaS B2B, SaaS B2C, Marketplace, API, Internal Tool, Mobile App, Platform, Other)"
- "¿En qué industria o vertical opera tu proyecto? (Ej: Healthcare, FinTech, Education, E-commerce, Manufacturing, etc.)"
- "¿En qué fase se encuentra actualmente tu proyecto? (Idea, MVP en desarrollo, Beta, Operando, Escalando)"

### Sección B: Contexto Adicional Profundo

B1. Restricciones y Limitaciones:
- "¿Existen restricciones técnicas específicas (legado, compatibilidad, infraestructura)?"
- "¿Hay requerimientos regulatorios o de cumplimiento que debamos cumplir?"
- "¿Qué limitaciones de presupuesto, tiempo o recursos tenemos?"
- "¿Hay stakeholders clave con poder de veto o requisitos especiales?"

B2. Contexto del Equipo:
- "¿Qué experiencia previa tiene tu equipo con proyectos similares o en esta industria?"
- "¿Cuáles son las fortalezas y debilidades técnicas del equipo actual?"
- "¿Hay preferencias tecnológicas o aversiones basadas en experiencia previa?"
- "¿Qué roles clave faltan en el equipo actual?"

B3. Ecosistema y Mercado:
- "¿Quiénes son los principales competidores o alternativas existentes?"
- "¿Qué tendencias del mercado afectan tu proyecto?"
- "¿Hay cambios tecnológicos o regulatorios en el horizonte?"

B4. Urgencia y Timing:
- "¿Hay algún deadline crítico que debamos priorizar?"
- "¿Por qué este momento es el momento adecuado para este proyecto?"
- "¿Qué ventana de oportunidad estamos tratando de aprovechar?"
- "¿Qué consecuencias hay si no actuamos ahora?"

B5. Métricas de Éxito:
- "¿Cómo se verá el éxito de este proyecto en 6 meses? ¿En 2 años?"
- "¿Qué métricas de negocio son críticas para los stakeholders?"
- "¿Qué indicadores técnicos medirán el progreso?"
- "¿Cuál es el ROI esperado y cómo lo mediremos?"

## Validación de Comprensión del Proyecto

### Confirmación Final del Agente

Después de recibir toda la información, el agente responderá:

"Entiendo que tu proyecto **[nombre]** es **[descripción]** en la industria de **[industria]**. Es un proyecto tipo **[tipo]** actualmente en fase **[estado]**. 

Contexto adicional identificado:
- Restricciones principales: [restricciones técnicas, regulatorias, presupuesto]
- Contexto del equipo: [experiencia, fortalezas, roles faltantes]
- Ecosistema: [competidores, tendencias, oportunidad de mercado]
- Urgencia: [timing crítico, ventana de oportunidad]
- Métricas de éxito: [indicadores clave de negocio y técnicos]

¿Es correcto este entendimiento completo?"

### Checkpoint de Validación Automática

Antes de continuar al siguiente paso, el agente debe generar:

#### Reporte de Validación de Contexto
- Completitud: ¿Se recopiló toda la información crítica?
- Claridad: ¿Hay ambigüedades que necesitan resolución?
- Coherencia: ¿La información es consistente entre sí?

#### Opciones de Mejora y Recomendaciones
- Información faltante: ¿Qué datos adicionales serían útiles?
- Áreas de riesgo: ¿Qué contextos requieren mayor exploración?
- Conexiones estratégicas: ¿Cómo este contexto informa la visión?
- Próximos pasos: Recomendaciones para BOOT-GUIDE-0001

El agente debe presentar este reporte y esperar confirmación antes de proceder.

## Uso de Plantillas Disponibles

### Plantilla Principal para Documentación del Proyecto

Usa la plantilla de contexto del proyecto para crear el primer documento de tu producto. La plantilla incluye:
- Secciones predefinidas para todas las categorías de contexto
- Espacios para restricciones, equipo, ecosistema, urgencia y métricas
- Sección de validación de contexto con recomendaciones

Cómo usar la plantilla:
1. Completa cada sección usando la información recopilada en el diálogo
2. Usa la sección de validación para confirmar completitud y coherencia
3. Este documento será la base para toda la documentación de tu proyecto

### Beneficios de Usar Plantillas

- Consistencia: Mismo formato en todos los documentos
- Profesionalismo: Estructura validada por liderazgo
- Eficiencia: No empezar desde cero
- Calidad: Secciones probadas que conectan con audiencias
- Mantenimiento: Actualizaciones fáciles

### Ubicación del Documento Generado

- **Documento**: Contexto del Proyecto (primer documento de tu producto)
- **Ubicación**: `documentation/executive/context/project-context.md`
- **Propósito**: Documento permanente que servirá como base para toda la documentación de tu proyecto
- **Justificación**: Es el fundamento estratégico para todos los documentos subsiguientes de tu producto
